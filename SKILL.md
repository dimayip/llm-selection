# 大模型选型 Skill

根据用户的具体需求和约束条件，系统化推荐最合适的大语言模型，输出结构化选型报告，包含成本预估和落地建议。

## 工作流程

### Step 1：收集需求信息

通过提问或从用户输入中提炼以下关键信息（未提供的字段主动询问）：

| 字段 | 必问 | 说明 |
|------|------|------|
| 使用场景 | ✅ | 聊天/代码/创作/推理/多模态/客服/文档处理/Agent工具调用等 |
| 预算范围 | ✅ | 每月 API 成本预算，或"尽量省" |
| 数据合规 | ✅ | 数据是否能出境（国内业务必须国内模型） |
| 上下文长度 | 可选 | 需要处理的最大文本长度（token 数） |
| 并发/QPS | 可选 | 预计每秒请求数 |
| 部署偏好 | 可选 | 云端 API / 可本地部署 / 必须本地部署 |
| 多模态需求 | 可选 | 是否需要图像/音频/视频理解 |
| 中文要求 | 可选 | 中文内容占比，是否需要优秀的中文理解/生成 |

**场景关键词映射（帮助用户快速描述需求）**：
- 聊天机器人 → 多轮对话、低延迟
- 内容创作 → 长文写作、文风控制
- 代码助手 → 代码生成、解释、调试
- 数据分析 → 数字理解、逻辑推理
- 文档处理 → 长文档摘要、信息提取
- 智能客服 → 低延迟、高并发
- 知识问答 → 事实准确性、知识广度
- Agent/工具调用 → 结构化输出、Function Calling
- 翻译 → 多语言、领域术语
- 图像理解 → 图文问答、OCR

### Step 2：硬约束初筛

根据硬约束过滤模型列表，不满足的直接排除：

1. **数据合规**：国内业务（数据不能出境）→ 仅保留国内模型（混元/通义/文心/DeepSeek/GLM/Kimi/豆包等）
2. **上下文长度**：保留 `max_context >= 用户需求` 的模型
3. **部署方式**：
   - 必须云端 API → 排除仅开源可本地部署的模型
   - 必须本地部署 → 仅保留有开源权重的模型
4. **多模态**：有图像/音频需求 → 保留 `multimodal=true` 的模型

初筛完成后，候选模型通常剩 3-8 个。

参考 `references/model_zoo.md` 获取完整的模型数据。

### Step 3：评分排序

对候选模型用加权评分矩阵打分（1-5 分），计算总分：

```
总分 = 推理能力×30% + 多模态×25% + 上下文能力×20% + 响应速度×15% + 成本效益×10%
```

**各项评分标准**：

| 维度 | 权重 | 5分 | 4分 | 3分 | 2分 | 1分 |
|------|------|-----|-----|-----|-----|-----|
| 推理能力 | 30% | MMLU>85, 专业基准顶尖 | MMLU 80-85 | MMLU 75-80 | MMLU 70-75 | MMLU<70 |
| 多模态 | 25% | 图文音视频全支持 | 图文音支持 | 图文支持 | 仅文本+图像 | 仅文本 |
| 上下文 | 20% | ≥128K | 32-128K | 8-32K | 4-8K | <4K |
| 响应速度 | 15% | 首token<500ms | 500ms-1s | 1-2s | 2-5s | >5s |
| 成本效益 | 10% | 极低（$0.1/1M以下） | 低（$0.1-1/1M） | 中（$1-5/1M） | 高（$5-15/1M） | 极高（>$15/1M） |

> **注意**：以上为通用标准，针对特定场景可调整权重。例如代码场景可提高"推理能力"权重，多模态场景可提高"多模态"权重。

### Step 4：输出选型报告

输出结构化选型报告，格式如下：

```
## 大模型选型报告

### 📋 需求摘要
- 使用场景：XXX
- 预算范围：XXX
- 数据合规：XXX
- 其他约束：XXX

### 🏆 推荐模型（Top 3）

#### 🥇 首选：[模型名称]
- 综合评分：X.X/5.0
- 推荐理由：[核心优势与场景匹配度说明]
- 预估月成本：[具体金额或区间]
- 优势：[3-5 个关键优势]
- 注意事项：[2-3 个需要注意的点]

#### 🥈 备选：[模型名称]
（同上格式）

#### 🥉 性价比之选：[模型名称]
（同上格式）

### 💰 成本对比
[表格：各候选模型的月成本对比，基于典型调用量]

### 🚀 落地建议
1. 小流量验证方案（A/B 测试设计）
2. Fallback 策略（主模型不可用时的备用方案）
3. 速率限制注意事项
4. 内容审核策略（如有需要）
```

### Step 5：成本估算

当用户关心成本时，调用 `scripts/cost_estimator.py` 或手动计算：

**成本估算公式**：
```
月成本（API）= 日均调用次数 × 30
    × (平均输入 token 数 × 输入单价 + 平均输出 token 数 × 输出单价)
    / 1,000,000
```

**自部署成本估算**：
```
月成本（自部署）= GPU 租赁费 + 电费 + 运维人力
示例：8×A100 租赁约 ¥20-30万/月
```

如用户提供调用量数据，直接计算；如未提供，给出不同调用量档位（1万/10万/100万次/天）的成本区间。

## 重要注意事项

1. **模型迭代快**：如用户问的模型在 `model_zoo.md` 中不存在，通过 `web_search` 查询最新信息后补充回答，并注明数据来源和日期
2. **价格变动**：API 价格可能调整，给出价格时注明"参考价，以官网为准"
3. **基准测试局限性**：不同 benchmark 结果差异大，优先参考用户在意的真实场景表现，不要只依赖 MMLU 等通用基准
4. **混合方案**：复杂场景可推荐混合方案（如简单 query 用低成本模型，复杂推理用高端模型）
5. **Fallback 必须提醒**：务必在报告中提醒用户设置备用模型，避免单点依赖
6. **国内模型优先**：当用户明确是国内业务，优先推荐国内模型（数据合规、延迟更低、中文更好）
7. **标注不确定性**：对不确定的信息（如某模型的最新价格），明确标注"需核实"
